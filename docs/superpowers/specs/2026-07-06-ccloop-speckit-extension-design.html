§0 What happened this session

Version gap, fixed

A project's .specify/ is a version-pinned snapshot from init time — global CLI upgrades never rewrite it. Re-materialized the project at 0.12.4 (specify init --here --force). Constitution, all four extensions, and every command-skill verified intact; backups kept.

The jump helped

0.12.4 adds a workflow engine (while/do-while/shell/gate/command/fan-out…) and a bundle primitive. The loop becomes a native workflow YAML — and fail-closed is native: a bash step exiting non-zero halts the run.

2 · The ccloop workflow — flowchart (specify workflow run ccloop)

Opus is billed only at arm / derive / reflect (thinking) plus review; the high-volume dispatch runs free on local hardware; the judge is a different model family; every harness step is bash that fails closed (non-zero exit HALTS the run). One human touchpoint: the done-gate.

§1 Locked decisions

  1. Pure extension — drop the standalone Claude Code plugin + marketplace form.
  2. Own the implement phase — consume tasks.md, execute each task on local models, judge-gate, Opus-review.
  3. Pluggable agent-CLI adapters (ralph-style); OpenCode + local models default; cross-family invariant maps onto configured adapters.
  4. Progress + feedback layer from aide (per-task DoD, monotonic progress, trade-offs log); skip vision/roadmap — spec.md/plan.md cover that.
  5. Approach C (hybrid) — tasks.md stays frozen/PROTECTED; run-state lives in a separate ID-keyed ledger. Resolves the ralph tension where the loop would mutate tasks.md.

§4 Packaging — bundle = manifest-of-references

Fable correction (B3): a 0.12.4 bundle is not a self-contained zip. bundle install reads only bundle.yml and resolves each component by id from core_pack / a catalog — the zip's own files are never extracted.
Install (LOCKED) — one command, straight from the repo URL:
specify extension add ccloop --from <repo-url>     # --from is a real 0.12.4 flag ✓

The extension ships its own workflow.yml (.specify/extensions/ccloop/workflow/workflow.yml). On the first /speckit.ccloop.run, the command self-registers it if absent — specify workflow add …/workflow.yml (accepts ID/URL/path ✓) — then specify workflow run ccloop. One install command; workflow registration is automatic + idempotent. A bundle.yml still versions the pair and can back a published catalog later. Component layout:

§6 The loop as a spec-kit workflow

Declarative YAML orchestration; every safety gate is a bash shell step whose non-zero exit halts the run. Corrected against the engine source (do-while, .output.data. paths, literal cap, fail-closed assert, detach+poll).

steps:
  - id: arm            # resolve feature, arm loop, cross-family preflight
    type: shell
    output_format: json                 # so steps.arm.output.data.feature resolves
    run: "bash .../state.sh arm --feature '{{ inputs.feature }}' --json"
  - id: freeze   {type: shell, run: ".../harness/freeze.sh"}      # hash-pin spine, fail-closed
  - id: derive   {type: shell, run: ".../contract-derive.sh"}      # tasks.md+plan.md → contract
  - id: loop
    type: do-while     # NOT while: a while checks its condition BEFORE iter 1,
                     # when loop_status has never run → falsy → body never runs (B1)
    condition: "{{ steps.loop_status.output.data.open > 0 }}"   # note .output.data (B2)
    max_iterations: 20   # LITERAL int only (B5); guards.sh owns the durable cap
    steps:
      - id: loop_status  {type: shell, output_format: json, run: ".../progress-status.sh --json"}
      - id: dispatch     # detach-start + poll — a direct dispatch WILL exceed the 300s cap (R1)
        {type: shell, run: ".../dispatch.sh next --detach --no-op-if-closed"}
      - id: judge        {type: shell, run: ".../judge.sh next --detach"}   # cross-family + sandbox
      - id: gate         {type: shell, run: ".../harness/gate.sh"}          # hash-verify + tests
      - id: record       {type: shell, run: ".../progress-lint.sh record"} # monotonic status
  - id: assert_closed  # FAIL-CLOSED: cap-exhaustion must not reach the gate as "success" (B4)
    {type: shell, run: ".../progress-status.sh --assert-closed"}
  - id: human_done_gate          # the ONE human gate; PAUSES if non-TTY → resume to sign off
    type: gate
    message: "All tasks judge-pass. Review debt below, then approve to close."
    show_file: "specs/{{ steps.arm.output.data.feature }}/ccloop/debt.md"
    options: [approve, reject]
    on_reject: abort
  - id: signoff  {type: shell, run: ".../done-gate.sh signoff"}   # records human sign-off (G3)
  - id: reflect  {type: shell, run: ".../reflect.sh run"}          # offline, quarantined candidates
Engine facts (Fable-verified): fail-closed is native (non-zero → step FAILED → run FAILED; continue_on_error is opt-in). The engine truly re-iterates the body and re-evaluates the condition against a re-run loop_status. Resume re-runs the whole do-while with a fresh cap → guards.sh owns the durable bound; progress.md idempotency makes re-runs safe. The 300s cap also hits judge and sandboxed tests → detach+poll everywhere heavy.

§7 Status ladder & adapters

Per task ID in progress.md (schema-pinned in state.sh, G2; role-scoped):

pending→ dispatched→ implemented→ judge-pass| judge-fail| judge-uncertain →debt→ human-signed

Implementer may only reach implemented; the cross-family judge owns the judge-* verdicts (uncertain opens a debt row, never vanishes); the human done-gate owns closure.

Pluggable adapters

get_agent_cli_kind() normalizer (bash-3.2-safe) → opencode|claude|codex|copilot|unsupported; per-CLI invocation shapes; inline the iterate contract for print-mode CLIs. Cross-family invariant holds — assert_cross_family(implementer, judge); Opus is never a local implementer. Config precedence: defaults → ccloop-config.yml → .local → SPECKIT_CCLOOP_* env → args.

§10 Guardrails — G1–G8 preserved, + G9

G1one injected memory (lessons.md only; build-context denylist gains specs/*/ccloop/**)   G2schema-pinned state   G3cadence proposes, humans promote + sign off   G4mechanical lessons cap, fail-closed   G5additive only   G6frozen calibration is the arbiter   G7retention split   G8deterministic telemetry authorship

G9tasks.md immutability — the loop never writes tasks.md; freeze/gate hash-verify it and progress-lint keeps completion in progress.md. This is the guard that makes ralph's pattern safe here.
